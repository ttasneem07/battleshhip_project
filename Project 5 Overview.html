Read the Software Requirements

Battleship is a two player game of strategy and luck. Each player deploys a fleet of ships on their board, and the goal of the game is to sink all of the other player’s ships before they sink your ships! The game of Battleship dates back to the 1800’s. In 1967, the popular Battleship board game was released by Milton Bradley, who subsequently released an electronic version of the game in 1977.

In the Battleship game, each player’s board is divided into a grid of squares. Play begins with each player arranging their ships on their board, which is not visible to the other player. The ships vary in size and occupy one or more squares in a straight line, either vertically or horizontally. The players then take turns “firing” at a square on the other player’s board. If the square contains a ship or part of a ship, the ship has been “hit.” When all parts of a ship have been hit, it is sunk. Gameplay ends when all of the ships in one player’s fleet have been sunk. The other player wins!

In our version of the Battleship game, a human player plays against the computer. Each player has the following ships that occupy the given number of squares on the player’s board:

  • Submarine (1 Square)
  • Destroyer (2 Squares)
  • Cruiser (3 Squares)
  • Battleship (4 Squares)

An initial setup stage allows the human player to place their fleet of ships. Ships are placed one at a time and can be placed with either a horizontal (default) or vertical orientation. The human player selects the upper left coordinate of where they want their ship to be placed. If the ship cannot be placed in the location due to another ship or the edge of the board, an error message is displayed.

After all the ships are placed, gameplay begins automatically. The human player gets to make the first move by clicking on one of the computer’s squares. The computer board is updated to show the move. A white circle indicates a miss, and a red circle indicates a hit. We’re providing you with a simple game AI for the computer player, who makes the second move. After the computer’s move, the human board is updated with a white circle indicating a miss or a red circle indicating a hit. Gameplay continues, alternating between the human and the computer player.

The game ends when either the human player has sunk all of the computer’s ships (a win) or the computer has sunk all of the human player’s ships (a loss). The human player is informed of the result of the game with a message below the game boards.

Program Execution - Testing Mode

In testing mode, we must provide a seed to our random number generator so that we may expect the same results with each execution of our program. Random numbers in Java are not truly random. Numbers are created using a repeatable algorithm that mimics random behavior. This algorithm is seeded with an initial value to start the generation of random numbers; typically the current system time in milliseconds. The random number generation algorithm is such that if provided the same seed, the numbers generated by calls to nextInt() will always occur in the same sequence. We will use this property to help test our program. If the user provides a seed as a command line argument, then the program will run in test mode. A sample execution is:

1

   $ java -cp bin BattleshipGUI 1

Design Your Software

This project is broken down into seven classes and uses the Model-View-Controller (MVC) design pattern.1

The Graphical User Interface, which makes up the View-Controller part of the program, is provided to you in BattleshipGUI.java and Grid.java. The “artificial intelligence” for the computer’s play is provided in BattleshipAI.java. The Battleship Model class is provided in Battleship.java. Do not change the four provided classes.

You will write the other three Model classes: Ship, Square, and Board.

Shown below is a complete UML Class Diagram that shows all of the constants, instance variables, constructors, and methods in the seven classes. It also illustrates the relationships between the four classes followed by an explanation of these relationships.

Model Classes: The Battleship class has an “aggregate relationship” with the Board class because it contains a Board object. Likewise, the Board class has an aggregate relationship with the Square and Ship classes because it contains objects of those classes. The Battleship class has a “dependency relationship” with the BattleshipAI class because it depends on BattleshipAI. The BattleshipAI class has a dependency relationship with the Board class and the Square class has a dependency relationship with the Ship class.

View-Controller Class: There is a dependency relationship between the BattleshipGUI class and the Battleship class, an aggregate relationship between the BattleshipGUI class and the Grid class, and dependency relationships between the Grid class and both the Board and BattleshipGUI classes.

Notice that while the View-Controller class depends on Model classes, the Model classes are not dependent on the View-Controller class. This makes it very easy to change to a different View-Controller for the Battleship game.

Our UML diagram has the following notation notation:

  • A - in front of a name means private.
  • A + in front of a name means public.
  • A ~ in front of a name means default.
  • An underlined name means static.

Implement Your Software

The model code that you will be creating by implementing the classes below keeps track of information about the battleship game. This information is used by the GUI to display the current state of the game. Also, the mouse clicks on the GUI generate method calls to the model that need to be implemented correctly in order for the game to proceed as required. Therefore, it is extremely important that you implement every method of every class exactly as we specify in the following class descriptions. You are encouraged to implement other private methods to simplify your code.

The Battleship game you are building will be broken down into seven classes—four classes that are provided and three class that you will write. Start your program by implementing and testing the Ship class, then implement and test the Square class, and finally, implement and test the Board class.

  • Provided classes (Do not change these classes.):
    • BattleshipGUI.java - The Graphical User Interface (GUI) the player uses to play the game.
    • Grid.java - Used by the GUI to represent and allow interaction with the computer and human player grids of squares, some of which contain a ship or part of a ship.
    • Battleship.java - Manages the Battleship game play.
    • BattleshipAI.java - Provides the AI for the computer player.
  • Classes you will write:
    • Ship - Represents a single ship.
    • Square - Represents a square that may contain a ship or part of a ship.
    • Board - Used by the Battleship class to represent and allow interaction with the computer and human player boards that are made up of squares, some of which contain a ship or part of a ship.

Test Your Software

To increase confidence that your Battleship program works correctly, you will implement automated unit tests and execute system test cases on your program. Unit testing will focus on testing the methods of the model classes. System testing will be used to ensure that all your classes work together with the provided GUI class.

System Testing

Complete the provided system test plan - see the directions below about how to access it. You will add five test cases. You will also run all of the test cases and fill in the actual results, which should match the expected results.

We have provided three test cases for you as examples of how to write your test cases. The first test case tests the initial GUI. The second test case tests the placement of Ships on the Human Board. The third test case tests the entire game play process where the human wins. The five tests that you add may test error conditions, for example, a different case of placing a ship incorrectly than the one provided, firing at the same square twice, etc. as well as a partial and/or entire game play.
